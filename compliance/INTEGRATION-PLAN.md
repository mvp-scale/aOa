# Compliance Integration Plan

> **Transient working document.** Source of truth for closing the gaps in `compliance/REPORT.md`. Every observed field is one row. We march down the rows and check `Tested = Y` when each is verified end-to-end.

## Lifecycle

This file is **temporary**. It exists to drive the work from "current state" to "fully integrated." When every row reaches `Tested = Y` and `compliance/REPORT.md` shows 100% integration coverage, this file is **deleted**.

The durable artifacts are:

| File                              | Lifecycle                                                |
|-----------------------------------|----------------------------------------------------------|
| `CONTRACT.md` (per surface)       | Permanent. Living spec of the integration contract.      |
| `REPORT.md` (top-level)           | Permanent. Auto-regenerated by the compliance test.      |
| `versions/v{X}-observed/*` (per surface) | Permanent. One snapshot per Claude Code version.    |
| `INTEGRATION-PLAN.md` (this file) | **Transient** — removed when all rows are complete.      |

If we need a similar plan in the future (e.g., a new Claude Code version introduces a wave of drift), a fresh INTEGRATION-PLAN.md is written then deleted again. It is **never** the permanent home of compliance state.

## Philosophy

Two distinct concepts:

| Concept       | Definition                                                         | Compliance impact |
|---------------|--------------------------------------------------------------------|-------------------|
| **Integration** | Parser reads the field, surfaces it on the canonical event       | **Required** — anything else is a silent drop |
| **Processing**  | A downstream consumer (learner, dashboard, telemetry) uses it    | **Optional** — driven by goal alignment |

**Default target for every row = `[READ + ...]`.** The work is deciding `USED` vs `HOLD` per row.

| Marker          | Meaning                                                                        |
|-----------------|--------------------------------------------------------------------------------|
| `[READ + USED]` | integrated and processed downstream                                            |
| `[READ + HOLD]` | integrated, surfaced on canonical event, no consumer yet (compliant + parked)  |
| `[SKIP]`        | intentionally not integrated, with documented reason (rare)                    |
| `[DROPPED]`     | currently neither read nor surfaced — silent drop (the gap we are closing)     |
| `[IGNORED]`     | entire event type unhandled (subset of DROPPED at type level)                  |

## Execution order

Two tracks, executable independently:

### Track A — Integration (parser + ports.SessionEvent)

Order chosen so each step is independently testable and committable. Each lands as one PR.

1. **S1.F1–F4** (`toolUseResult` discriminated union) — biggest refactor; introduces a new ports type
2. **S1.E2–E12** (envelope context fields) — small parser change + ports.SessionEvent extension
3. **S1.T4–T7** (new event type translators) — wire each through `reader.go`
4. **S1.U2–U6, S1.M2–M6, S1.S2** (usage + message-level + away_summary subtype) — pure additions
5. **S2.P2** (statusline live capture + new path consumption) — one-shot capture procedure

### Track B — Processing (downstream consumers)

After Track A lands the data on canonical events, downstream consumers opt in. These are smaller and isolated:

- **S1.F2 `Edit.userModified`** → learner (G5 — direct quality signal)
- **S1.F1 `Bash.interrupted`** → learner (G5 — negative signal)
- **S1.F3 Agent recap fields** → telemetry rollup (G6 — replace JSONL reconstruction)
- **S1.U2 `cache_creation` nested** → dashboard cache analysis (G6)
- **S1.U3 `server_tool_use`** → dashboard burn rate (G6 — currently miscounts)
- **S1.M4 `stop_reason`** → completion-quality metric (G5/G6)
- **S1.S2 `away_summary.content`** → bigram extraction (G5)
- **S1.E2 `cwd`, S1.E3 `gitBranch`** → telemetry context (G6)
- **S1.E9 `sourceToolAssistantUUID`** → tool correlation refactor (G1 — replaces synthesized link)
- **S1.E12 `permissionMode` + S1.T4 `permission-mode` event** → permission-mode-aware metrics (G5)

## Tested column

`Tested = Y` requires **all three**:

1. Parser unit test exists for the field/shape and passes
2. Field appears on canonical event in compliance test (move from `knownUnconsumed*` to `consumed*` map)
3. If processed: downstream consumer's behavior verified (parity test, dashboard render, etc.)

When all rows reach `Tested = Y`, **REPORT.md will show 100% integration coverage** and the compliance test's "BREAKING DRIFT" gate becomes meaningful (anything Claude Code adds outside this plan = blocker).

## Maintenance

When Claude Code ships a new version:
1. Run `go test -tags compliance ./compliance/...` and read REPORT.md.
2. New types/fields surface as drift warnings.
3. **Add a row to this plan** for each. Decide Target + Value with goal alignment.
4. Execute Track A (integrate), then optionally Track B (process).
5. Re-run; report should re-stabilize at 100% integration.
